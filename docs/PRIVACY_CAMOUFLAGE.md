# وضع الخصوصية والتمويه المشروع

## 1. الهدف

يهدف هذا المستند إلى تعريف مفهوم “التمويه المشروع” داخل النظام. المقصود بالتمويه هنا هو حماية المحتوى الأساسي، التفضيلات الشخصية، السجلات الحساسة، والإعدادات الأمنية من الوصول غير المصرح به أو الفضول غير المبرر.

لا يعني التمويه إخفاء نشاط ضار، أو تجاوز أدوات الأمان، أو خداع أنظمة الحماية، أو مخالفة قوانين أو سياسات التطبيقات والمنصات.

## 2. المبدأ الأساسي

يجب أن يكون النظام مفهومًا وشفافًا لصاحبه فقط، بينما لا يكشف تفاصيله الحساسة لأي شخص آخر بدون تحقق من هوية المستخدم.

بمعنى:

- المستخدم المصرح له يرى الحقيقة الكاملة بعد المصادقة.
- أي شخص غير مصرح له يرى واجهة محدودة أو عامة لا تكشف المحتوى الأساسي.
- لا يتم عرض الأسرار أو السجلات الحساسة إلا بعد تحقق قوي مثل Face ID أو البصمة أو رمز الجهاز أو المصادقة الثنائية.

## 3. ما هو مسموح في التمويه المشروع

### 3.1 واجهة عامة غير حساسة

يمكن للنظام أن يعرض واجهة بسيطة لا تكشف الغرض الكامل، مثل:

- صفحة ملاحظات عامة.
- لوحة مهام يومية.
- صفحة إعدادات محدودة.
- وصف عام مثل “مساعد شخصي” بدل عرض تفاصيل الحماية كاملة.

### 3.2 إخفاء التفاصيل الحساسة حتى المصادقة

قبل التحقق من هوية المستخدم، يجب إخفاء:

- التفضيلات الشخصية الدقيقة.
- سجل الإجراءات الأمنية.
- بيانات التطبيقات والحسابات.
- مفاتيح API أو رموز الوصول.
- إعدادات الشبكة الحساسة.
- نتائج الفحص الأمني.

### 3.3 تشفير المحتوى الحساس

يجب تخزين المحتوى الحساس مشفرًا، ولا يتم فك تشفيره إلا بعد تحقق المستخدم.

يفضل استخدام حلول النظام الرسمية مثل:

- Apple Keychain.
- Android Keystore.
- Windows Credential Manager.
- مفاتيح محلية محمية بالنظام.

### 3.4 وضع العرض الآمن

عند عدم وجود مصادقة قوية، يعرض النظام معلومات عامة فقط، مثل:

```text
النظام يعمل بشكل طبيعي.
لا توجد تنبيهات عامة.
يرجى المصادقة لعرض التفاصيل.
```

### 3.5 حماية الإشعارات

يجب ألا تعرض الإشعارات تفاصيل حساسة على شاشة القفل.

مثال آمن:

```text
يوجد تنبيه خصوصية يحتاج مراجعتك.
```

بدلًا من:

```text
تطبيق محدد يحاول الوصول إلى موقعك وجهات اتصالك.
```

## 4. ما هو غير مسموح

لا يسمح للنظام بأي من الآتي:

- إخفاء برمجيات ضارة.
- تجاوز أدوات مكافحة الفيروسات أو أنظمة الحماية.
- تعطيل سجلات النظام أو حذفها لإخفاء نشاط ما.
- التلاعب بتطبيقات خارجية بطريقة تخالف شروط الاستخدام.
- جمع بيانات الآخرين دون موافقة.
- إنشاء آليات مراقبة خفية.
- استخدام التمويه لخداع المستخدم نفسه أو منعه من معرفة ما يحدث.

## 5. مستويات الظهور

### المستوى 0: غير مصادق

ما يظهر:

- واجهة عامة.
- معلومات غير حساسة.
- لا توجد أسرار.
- لا توجد نتائج فحص مفصلة.

ما لا يظهر:

- السجلات.
- الإعدادات الحساسة.
- التفضيلات الشخصية.
- بيانات الحسابات.

### المستوى 1: مصادقة عادية

وسائل المصادقة:

- رمز الجهاز.
- جلسة موثوقة قصيرة.

ما يظهر:

- ملخصات عامة.
- مهام يومية.
- إعدادات منخفضة الخطورة.

### المستوى 2: مصادقة قوية

وسائل المصادقة:

- Face ID.
- بصمة الإصبع.
- المصادقة الثنائية.

ما يظهر:

- تفاصيل الخصوصية.
- سجل التدقيق.
- نتائج الفحص الأمني.
- إعدادات التطبيقات الحساسة.

### المستوى 3: مصادقة قوية مع تأكيد صريح

يستخدم للإجراءات الحساسة جدًا مثل:

- حذف بيانات.
- تغيير إعدادات أمان رئيسية.
- إدارة مفاتيح أو رموز وصول.
- تعديل صلاحيات تطبيقات مهمة.

## 6. تصميم الواجهة المموهة بأمان

يجب أن تكون الواجهة العامة صادقة لكنها غير كاشفة.

مثال مقبول:

```text
مساعد شخصي لإدارة المهام والإعدادات.
```

مثال غير مقبول:

```text
تطبيق فارغ تمامًا بينما يعمل في الخلفية دون علم المستخدم.
```

الفرق أن التمويه المشروع يحمي التفاصيل من غير المصرح لهم، لكنه لا يخدع صاحب الجهاز ولا يخفي نشاطًا ضارًا.

## 7. حماية أسماء الملفات والمجلدات

يمكن استخدام أسماء عامة للملفات الحساسة، بشرط ألا يكون الهدف خداع أدوات الأمان أو إخفاء برمجيات ضارة.

أمثلة مناسبة:

- `private-data.enc`
- `secure-notes.enc`
- `audit-log.enc`

أمثلة يجب تجنبها:

- أسماء توحي بأنها ملفات نظام وهي ليست كذلك.
- أسماء تهدف إلى تضليل أدوات الفحص.
- أسماء تستخدم لإخفاء نشاط غير مشروع.

## 8. سجل التدقيق مع الخصوصية

يجب أن يحتفظ النظام بسجل تدقيق، لكن دون كشف الأسرار.

مثال:

```text
تمت مراجعة إعدادات الخصوصية لتطبيق معين بعد مصادقة قوية.
```

بدلًا من تخزين تفاصيل حساسة مثل:

```text
اسم الحساب، الرمز، المفتاح، محتوى الرسائل.
```

## 9. سياسة الوصول إلى المحتوى الأساسي

لا يتم عرض المحتوى الأساسي إلا عند تحقق الشروط التالية:

1. الجهاز موثوق.
2. المستخدم مصادق عليه.
3. الجلسة لم تنتهِ صلاحيتها.
4. مستوى الإجراء يطابق مستوى المصادقة.
5. لا توجد محاولة وصول مريبة.

## 10. الاستجابة لمحاولات الوصول غير المصرح

عند محاولة فتح النظام دون صلاحية، يجب أن يتصرف النظام بهدوء:

- لا يعرض تفاصيل حساسة.
- لا يكشف سبب الرفض بتفاصيل زائدة.
- يعرض رسالة عامة.
- يسجل محاولة الوصول دون تخزين بيانات حساسة.

مثال:

```text
لا يمكن عرض هذه المعلومات الآن. يرجى المصادقة للمتابعة.
```

- عدم استخدام التمويه لإخفاء نشاط ضار.

## 12. النتيجة المطلوبة

النظام يجب أن يحمي محتواه الأساسي من غير المصرح لهم، ويجعل التفاصيل الحساسة متاحة فقط لصاحب النظام بعد تحقق قوي، مع الحفاظ على الشفافية الكاملة للمستخدم المصرح له وعدم بناء أي آليات ضارة أو مخالفة.
