# سياسة التعامل مع تطبيقات التواصل الاجتماعي والأجهزة

## 1. الهدف

يحدد هذا المستند حدود تعامل النظام مع تطبيقات التواصل الاجتماعي والأجهزة الشخصية، مع التركيز على الخصوصية، عدم التدخل غير المطلوب، وعدم الدخول المباشر إلى حسابات المستخدم أو أنظمة التطبيقات.

الفكرة الأساسية: النظام يساعد المستخدم في الفهم، التذكير، إعداد المحتوى، والتنبيه إلى المخاطر، لكنه لا يدخل إلى حسابات التواصل الاجتماعي ولا يتحكم بها مباشرة دون إذن صريح وطرق رسمية.

## 2. عدم الدخول إلى أنظمة التواصل الاجتماعي

لا يجب أن يقوم النظام بالدخول المباشر إلى حسابات أو أنظمة تطبيقات التواصل الاجتماعي مثل:

- Facebook.
- Instagram.
- WhatsApp.
- TikTok.
- X / Twitter.
- Snapchat.
- Google services.

### المسموح

- شرح إعدادات الخصوصية.
- اقتراح إعدادات مناسبة.
- إعداد نصوص منشورات أو رسائل.
- تنبيه المستخدم إلى مخاطر ظاهرة.
- تلخيص شروط وسياسات التطبيقات إذا وفرها المستخدم أو كانت متاحة رسميًا.
- تذكير المستخدم بمراجعة إعدادات الحساب.

### غير المسموح

- تسجيل الدخول نيابة عن المستخدم دون موافقة.
- تجاوز حماية التطبيقات.
- استخدام طرق غير رسمية للسيطرة على الحساب.
- إرسال رسائل أو نشر محتوى تلقائيًا دون موافقة.
- قراءة رسائل أو بيانات خاصة دون إذن واضح.
- مخالفة شروط استخدام التطبيقات.

## 3. عدم الارتباط مع Apple Watch

لا يجب أن يكون للنظام أي اعتماد أو ارتباط إجباري مع Apple Watch.

### القاعدة

- لا يستخدم النظام Apple Watch كوسيلة تحكم أساسية.
- لا يقرأ بيانات Apple Watch الصحية أو الشخصية.
- لا يعتمد على Apple Watch للمصادقة.
- لا يرسل إليها إشعارات حساسة إلا إذا اختار المستخدم ذلك لاحقًا وبشكل صريح.

إذا احتاج النظام إلى مصادقة، يجب أن يعتمد على الهاتف أو الحاسوب مباشرة، مثل Face ID أو البصمة أو رمز الجهاز أو المصادقة الثنائية.

## 4. معرفة تفضيلات المستخدم

يجب أن يكون النظام على دراية بما يفضله المستخدم وما لا يفضله، لكن بطريقة شفافة وقابلة للتعديل.

### أمثلة على التفضيلات

- نوع المحتوى المفضل.
- طريقة الصياغة المفضلة.
- المواضيع التي لا يريد المستخدم التعامل معها.
- التطبيقات التي يريد مراقبة إعداداتها فقط دون الدخول إليها.
- مستوى الخصوصية المطلوب.
- نوع التنبيهات المقبول.
- أوقات عدم الإزعاج.

### قاعدة مهمة

لا يتم استخدام التفضيلات للتلاعب بالمستخدم أو نشر محتوى دون علمه. التفضيلات تستخدم فقط لتحسين الاقتراحات والتنبيهات والمساعدة اليومية.

## 5. الإجراءات المشتبه بها أو الحساسة

أي إجراء يعتبر حساسًا أو يحتوي على شبهة يجب ألا يتم إلا بعد تحقق قوي.

### أمثلة الإجراءات الحساسة

- إرسال رسالة إلى شخص أو مجموعة.
- نشر منشور على حساب أو جروب.
- تعديل إعدادات الخصوصية.
- ربط تطبيق بحساب خارجي.
- مشاركة بيانات أو ملفات.
- تصدير سجل أو تقرير.
- تغيير إعدادات أمان.
- الوصول إلى بيانات مالية أو محاسبية.

### التحقق المطلوب

عند وجود إجراء حساس، يجب طلب:

1. Face ID أو بصمة أو مصادقة بيومترية متاحة.
2. المصادقة الثنائية عند توفرها أو عند ارتفاع مستوى الخطورة.
3. تأكيد صريح من المستخدم قبل التنفيذ.

## 6. فتح الهاتف بالرمز بدل Face ID

إذا تم فتح الهاتف باستخدام رمز المرور فقط بدل Face ID أو البصمة، يجب أن يتعامل النظام بحذر أعلى.

### الوضع المقترح: الواجهة المحدودة

عند فتح الجهاز بالرمز فقط، يمكن للنظام أن يعرض واجهة محدودة لا تكشف التفاصيل الحساسة.

ما يظهر:

- مهام عامة.
- واجهة بسيطة.
- إشعارات غير حساسة.
- رسالة مثل: “يرجى المصادقة لعرض التفاصيل”.

ما لا يظهر:

- بيانات الحسابات.
- تقارير المحاسبة.
- أرصدة العملاء.
- إعدادات الأمان التفصيلية.
- سجل التدقيق.
- نتائج الفحص الأمني.
- الرسائل أو المنشورات الجاهزة للنشر.

## 7. فتح التفاصيل بعد المصادقة القوية

لعرض البيانات الحساسة يجب أن تتم المصادقة القوية.

مثال:

```text
هذه المعلومات حساسة.
يرجى استخدام Face ID أو المصادقة الثنائية لعرض التفاصيل.
```

بعد المصادقة، يمكن عرض التفاصيل حسب مستوى الصلاحية.

## 8. الواجهة البديلة عند الاشتباه

إذا لاحظ النظام سياقًا غير طبيعي، مثل محاولة فتح بيانات حساسة بعد فشل Face ID أو فتح الجهاز بالرمز فقط، يمكن عرض واجهة بديلة آمنة.

### الهدف

حماية المعلومات الحساسة من أي شخص قد يعرف رمز الهاتف لكنه ليس المستخدم المصرح له.

### يجب أن تكون الواجهة البديلة

- غير كاشفة للتفاصيل الحساسة.
- صادقة وغير ضارة.
- لا تخدع المستخدم المصرح له.
- لا تحذف بيانات.
- لا تعطل السجلات.
- لا تنفذ إجراءات في الخلفية.

## 9. مراقبة السوشيال ميديا بدون دخول مباشر

يمكن للنظام مساعدة المستخدم في مراقبة الأمور العامة أو المسموحة دون الدخول إلى حساباته مباشرة.

### أمثلة مسموحة

- تذكير المستخدم بمراجعة إعدادات الخصوصية.
- تنبيه عند رغبة المستخدم في نشر محتوى قد يكون حساسًا.
- إعداد نص منشور قبل النشر.
- اقتراح أفضل وقت للنشر بناءً على تفضيلات المستخدم إذا توفرت بيانات مسموحة.
- مراجعة النص لغويًا أو تجاريًا قبل إرساله.

### غير مسموح

- مراقبة حسابات الآخرين دون موافقة.
- استخراج بيانات خاصة من المنصات.
- كسر قيود التطبيقات.
- استخدام أدوات خفية للنشر أو السحب أو التتبع.

## 10. قاعدة عدم التنفيذ التلقائي

في تطبيقات التواصل الاجتماعي، يجب أن تكون القاعدة أكثر صرامة:

النظام يمكنه أن يجهز ويقترح، لكنه لا ينشر ولا يرسل ولا يعدل دون موافقة صريحة.

مثال:

```text
تم تجهيز منشور مناسب لجروب الزبائن.
هل تريد مراجعته؟ نعم / لا

بعد المراجعة:
هل توافق على نشره؟ نعم / لا
```

## 11. الخصوصية حسب نوع البيانات

### بيانات منخفضة الحساسية

- ملاحظات عامة.
- تذكيرات غير مالية.
- مهام يومية عامة.

يمكن عرضها بعد فتح الجهاز بشكل عادي.

### بيانات متوسطة الحساسية

- مسودات منشورات.
- إعدادات تفضيلية.
- أسماء تطبيقات.

تحتاج مصادقة عادية أو جلسة موثوقة.

### بيانات عالية الحساسية

- بيانات محاسبية.
- أسماء عملاء وأرصدة.
- رسائل خاصة.
- إعدادات أمان.
- تقارير فحص.

تحتاج Face ID أو بصمة أو مصادقة ثنائية.

## 12. سجل التدقيق

يجب تسجيل كل محاولة وصول حساسة بطريقة آمنة.

السجل يحتوي على:

- وقت المحاولة.
- نوع البيانات المطلوبة.
- هل تمت المصادقة أم لا.
- هل تم عرض واجهة محدودة.
- هل تم تنفيذ أي إجراء.

ولا يحتوي على:

- كلمات مرور.
- رموز تحقق.
- محتوى رسائل خاصة.
- مفاتيح API.

## 13. النتيجة المطلوبة

النظام يجب أن يحترم حدود المستخدم بوضوح:

- لا يدخل إلى حسابات التواصل الاجتماعي مباشرة.
- لا يرتبط بـ Apple Watch.
- يتعلم تفضيلات المستخدم لتحسين المساعدة فقط.
- يطلب Face ID والمصادقة الثنائية عند الشبهة أو الحساسية.
- يعرض واجهة محدودة إذا فُتح الهاتف بالرمز فقط أو في سياق غير موثوق.
- لا ينشر أو يرسل أو يغير شيئًا دون موافقة صريحة.
