# سياسة المصادقة ومنع الاعتماد على رقم الهاتف أو كود SMS

## 1. الهدف

يحدد هذا المستند سياسة المصادقة داخل النظام، مع التأكيد على عدم الاعتماد على رقم الهاتف أو كود الرسائل النصية SMS كوسيلة أساسية للتحقق من هوية المستخدم.

الهدف هو حماية المستخدم من مخاطر سرقة رقم الهاتف، تبديل شريحة SIM، اعتراض الرسائل، أو إجبار المستخدم على استخدام وسائل تحقق أقل أمانًا.

## 2. القاعدة الأساسية

لا يجب أن يعتمد النظام على:

- رقم الهاتف كوسيلة تحقق أساسية.
- كود SMS كوسيلة تحقق أساسية.
- مكالمة هاتفية آلية للتحقق.
- كود يصل عبر الهاتف كبديل وحيد للمصادقة.

يجب أن تكون المصادقة الأساسية عبر وسائل أقوى وأكثر أمانًا.

## 3. وسائل المصادقة المسموحة

### 3.1 المصادقة البيومترية

مسموح باستخدام:

- Face ID.
- بصمة الإصبع.
- المصادقة البيومترية التي يوفرها نظام التشغيل.

### 3.2 المصادقة الثنائية القوية

مسموح باستخدام:

- تطبيق مصادقة Authenticator App.
- مفتاح أمان مادي Security Key.
- Passkeys.
- رمز تحقق يتم توليده محليًا من تطبيق موثوق.

### 3.3 رمز الجهاز كعامل محدود

يمكن استخدام رمز الجهاز فقط كعامل مساعد أو احتياطي محدود، وليس كبديل كامل لعرض البيانات عالية الحساسية.

إذا تم فتح الجهاز بالرمز فقط دون Face ID أو بصمة، يجب تفعيل واجهة محدودة لا تعرض البيانات الحساسة.

## 4. وسائل غير مفضلة أو ممنوعة

### ممنوع كوسيلة أساسية

- SMS OTP.
- كود عبر مكالمة هاتفية.
- الاعتماد على رقم الهاتف وحده.
- روابط تسجيل دخول عبر رسالة نصية.

### مسموح فقط كاستثناء مؤقت عند الضرورة

يمكن السماح برقم الهاتف أو SMS فقط كخيار استرداد اضطراري، وبشروط صارمة:

- لا يكون مفعلًا افتراضيًا.
- يحتاج موافقة المستخدم.
- يستخدم فقط لاسترداد الحساب عند فقدان كل الوسائل الأخرى.
- يتم تسجيل استخدامه في سجل التدقيق.
- يتم تنبيه المستخدم إلى أنه أقل أمانًا.

## 5. الإجراءات الحساسة

أي إجراء حساس يجب أن يتطلب مصادقة قوية.

أمثلة الإجراءات الحساسة:

- عرض تقارير محاسبية.
- عرض أرصدة العملاء.
- إرسال رسالة أو نشر محتوى.
- تعديل إعدادات الخصوصية.
- ربط خدمة خارجية.
- تصدير ملفات.
- حذف بيانات.
- تغيير إعدادات الطباعة أو الأرشفة.
- السماح بعملية خلفية أو اتصال خارجي.

### التحقق المطلوب

يجب استخدام واحد أو أكثر من التالي حسب الخطورة:

- Face ID أو بصمة.
- Passkey.
- تطبيق مصادقة.
- مفتاح أمان.
- تأكيد صريح داخل التطبيق.

## 6. منع التطبيقات من فرض رقم الهاتف

عند التعامل مع تطبيقات أو خدمات خارجية، يجب أن يفضل النظام الخدمات التي تسمح بوسائل تحقق قوية وغير معتمدة على رقم الهاتف.

إذا طلب تطبيق خارجي رقم الهاتف أو كود SMS، يجب أن يقوم النظام بالتالي:

1. تنبيه المستخدم أن هذه الوسيلة أقل أمانًا.
2. اقتراح بدائل إن وجدت، مثل Passkey أو Authenticator App أو Security Key.
3. عدم حفظ رقم الهاتف أو كود SMS داخل النظام.
4. عدم قراءة رسائل SMS تلقائيًا.
5. عدم استخدام SMS كعامل وحيد للوصول إلى بيانات حساسة.

## 7. عدم قراءة رسائل SMS

لا يجب أن يقرأ النظام رسائل SMS أو أكواد التحقق تلقائيًا.

الأسباب:

- حماية الخصوصية.
- منع وصول النظام إلى رسائل شخصية.
- تقليل مخاطر إساءة استخدام أكواد التحقق.
- منع الاعتماد على قناة أقل أمانًا.

## 8. فتح الهاتف بالرمز فقط

إذا تم فتح الهاتف بالرمز فقط دون Face ID أو بصمة، يجب ألا يعرض النظام التفاصيل الحساسة.

يعرض فقط واجهة محدودة مثل:

```text
يرجى استخدام Face ID أو مصادقة قوية لعرض التفاصيل.
```

ولا يعرض:

- التقارير المالية.
- أرصدة العملاء.
- إعدادات الأمان.
- سجل التدقيق.
- نتائج الفحص.
- الملفات الحساسة.

## 9. إعدادات المستخدم

يجب أن يستطيع المستخدم تحديد سياسة المصادقة المفضلة:

```text
وسائل المصادقة المسموحة:
- Face ID: مفعّل
- بصمة: مفعّل عند توفرها
- Passkey: مفضل
- Authenticator App: مفضل
- Security Key: اختياري
- SMS: غير مسموح كوسيلة أساسية
- رقم الهاتف: غير مسموح كهوية أساسية
```

## 10. سجل التدقيق

يجب تسجيل محاولات المصادقة الحساسة دون تخزين الأسرار.

يسجل النظام:

- نوع المصادقة المستخدمة.
- وقت المحاولة.
- هل نجحت أو فشلت.
- نوع الإجراء المطلوب.
- هل تم تفعيل واجهة محدودة.

ولا يسجل:

- أكواد تحقق.
- رسائل SMS.
- كلمات مرور.
- مفاتيح خاصة.
- رموز وصول.

## 11. النتيجة المطلوبة

النظام يجب أن يرفض الاعتماد على رقم الهاتف أو كود SMS كوسيلة تحقق أساسية، وأن يستخدم وسائل أقوى مثل Face ID، البصمة، Passkeys، تطبيقات المصادقة، أو مفاتيح الأمان، مع عرض واجهة محدودة عند عدم توفر مصادقة قوية.
