# متطلبات النظام: مساعد حماية وتحكم شخصي

## 1. الهدف

بناء نظام مساعد ذكي شخصي يساعد مالك الأجهزة في إدارة الحماية، الخصوصية، الشبكة، التطبيقات، والمهام اليومية، مع ضمان أن أي إجراء مؤثر لا يتم إلا بعد التحقق من هوية المستخدم وموافقته الصريحة.

النظام يجب أن يعمل لصالح المستخدم فقط، وأن يوضح ما يفعله، ولماذا يفعله، وما الأثر المتوقع قبل التنفيذ.

## 2. مبدأ التحكم الشخصي

يجب أن يكون المستخدم هو صاحب القرار النهائي. الذكاء الاصطناعي يمكنه التحليل، الاقتراح، التنبيه، الترتيب، والشرح، لكنه لا ينفذ إجراءات حساسة أو مؤثرة دون موافقة صريحة.

### مسموح للنظام

- اقتراح أفضل إعدادات الخصوصية والأمان.
- تحليل المخاطر الظاهرة في التطبيقات أو الملفات أو الإعدادات.
- تنبيه المستخدم عند وجود صلاحيات زائدة أو اتصالات خارجية غير واضحة.
- مساعدة المستخدم في فهم سياسات التطبيقات وشروط الاستخدام.
- تنفيذ إجراءات مصرح بها بعد موافقة المستخدم.

### غير مسموح للنظام

- تجاوز سياسات التطبيقات أو المنصات.
- الوصول إلى حسابات أو بيانات لا يملكها المستخدم.
- استخراج بيانات مخفية بطريقة غير مصرح بها.
- تعطيل وسائل حماية التطبيقات أو كسر قيودها الأمنية.
- تنفيذ أوامر حساسة دون تحقق هوية وموافقة.

## 3. معرفة تفضيلات المستخدم

يجب أن يتعلم النظام تفضيلات المستخدم بطريقة شفافة وقابلة للتعديل.

### يجب أن يعرف النظام

- ما يحبه المستخدم.
- ما يكرهه المستخدم.
- التطبيقات التي يستخدمها بكثرة.
- مستويات الخصوصية التي يفضلها.
- أنواع الإشعارات التي يريدها أو لا يريدها.
- الأسلوب المفضل في الشرح: مختصر، تفصيلي، عملي، أو رسمي.
- الإجراءات التي يسمح بها دائمًا.
- الإجراءات التي تتطلب موافقة في كل مرة.
- الإجراءات الممنوعة نهائيًا.

### شرط مهم

يجب أن يتم تخزين هذه التفضيلات بطريقة واضحة، ويمكن للمستخدم مراجعتها أو تعديلها أو حذفها في أي وقت.

## 4. نموذج الصلاحيات

لا يتم اعتماد مفهوم “صلاحيات مطلقة بلا قيود”. البديل الآمن هو:

**صلاحيات مرتفعة مشروطة بالتحقق من هوية المستخدم وموافقته.**

### مستويات الصلاحيات المقترحة

#### المستوى 1: قراءة وإرشاد

- قراءة الملفات أو الإعدادات التي يختارها المستخدم.
- تقديم شرح أو توصيات.
- لا يوجد تنفيذ تلقائي.

#### المستوى 2: اقتراح وتنبيه

- تحليل المخاطر.
- اقتراح تغييرات.
- إعداد خطة تنفيذ.
- لا يتم التعديل إلا بعد الموافقة.

#### المستوى 3: تنفيذ محدود بموافقة

- تنفيذ إعدادات بسيطة بعد موافقة المستخدم.
- يشترط إظهار ملخص واضح قبل التنفيذ.

#### المستوى 4: تنفيذ حساس بتحقق قوي

يستخدم فقط للإجراءات الحساسة مثل:

- تعديل إعدادات أمان رئيسية.
- إدارة مفاتيح أو رموز وصول.
- تغيير إعدادات الخصوصية المهمة.
- ربط أو فصل حسابات.
- حذف بيانات أو تغيير صلاحيات تطبيقات.

يتطلب هذا المستوى:

- Face ID أو بصمة أو تحقق بيومتري متاح عبر نظام التشغيل.
- المصادقة الثنائية عند الحاجة.
- تأكيد صريح من المستخدم.
- تسجيل العملية في سجل تدقيق.

## 5. التحقق من هوية المستخدم

النظام يجب أن يتأكد أن الشخص الذي يعطي الأمر هو المستخدم المصرح له فقط.

### وسائل التحقق المقبولة

- Face ID أو بصمة الإصبع عند توفرها عبر نظام التشغيل.
- رمز مرور الجهاز.
- المصادقة الثنائية.
- تأكيد داخل التطبيق.
- جلسة موثوقة بزمن محدود.

### قاعدة إلزامية

أي وصول إلى معلومات حساسة أو تنفيذ إجراء حساس يجب ألا يتم إلا بعد تحقق هوية المستخدم.

## 6. الخصوصية والبيانات

### مبادئ الخصوصية

- لا يتم جمع بيانات غير ضرورية.
- لا يتم إرسال بيانات إلى خدمات خارجية دون سبب واضح وموافقة.
- لا يتم تخزين أسرار مثل كلمات المرور أو مفاتيح API داخل ملفات عامة.
- يجب توضيح أي اتصال خارجي قبل استخدامه.
- يجب إتاحة تعطيل الميزات غير الضرورية.

### بيانات حساسة تحتاج حماية خاصة

- كلمات المرور.
- رموز المصادقة.
- مفاتيح API.
- الرسائل الخاصة.
- الصور والملفات الشخصية.
- بيانات الموقع.
- جهات الاتصال.
- سجل التصفح أو النشاط.

## 7. فحص التطبيقات والملفات

يجب أن يساعد النظام في كشف المؤشرات التي قد تؤثر على الخصوصية أو الأمان، مثل:

- سكربتات تعمل تلقائيًا.
- اتصالات خارجية غير مبررة.
- صلاحيات زائدة.
- مكتبات غير ضرورية أو مشبوهة.
- ملفات إعداد تحتوي على أسرار.
- سياسات استخدام غير واضحة.
- إعدادات قد تقيّد المستخدم أو تجمع بياناته دون وضوح.

### حدود مهمة

النظام لا يستطيع ضمان كشف كل ما تخفيه التطبيقات المغلقة أو الخدمات الخارجية. يمكنه تحليل ما هو متاح للمستخدم قانونيًا وتقنيًا، وشرح المخاطر، واقتراح خطوات آمنة.

## 8. التطبيقات المدعومة

الهدف هو دعم كل التطبيقات التي يستخدمها المستخدم، لكن بشكل تدريجي وآمن.

### المرحلة الأولى

- إعدادات النظام الأساسية.
- المتصفح.
- البريد الإلكتروني.
- GitHub.
- Google Account.
- WhatsApp.
- Facebook.
- Instagram.

### طريقة الدعم

- شرح الإعدادات.
- اقتراح إعدادات خصوصية.
- إرشاد المستخدم خطوة بخطوة.
- تلخيص السياسات.
- التنبيه إلى الصلاحيات أو الروابط أو الإعدادات الخطرة.

### غير مسموح

- كسر حماية التطبيقات.
- تجاوز شروط الاستخدام.
- الوصول غير المصرح به إلى بيانات الغير.
- تشغيل مراقبة خفية.
- تعديل تطبيقات مغلقة بطرق غير مصرح بها.

## 9. الموافقة قبل أي إجراء

قبل تنفيذ أي إجراء، يجب عرض بطاقة موافقة تحتوي على:

- اسم الإجراء.
- سبب الإجراء.
- البيانات التي سيتم الوصول إليها.
- الصلاحيات المطلوبة.
- الأثر المتوقع.
- هل يمكن التراجع عنه أم لا.
- مستوى الخطورة.
- خيار الموافقة أو الرفض.

مثال:

```text
الإجراء: تقليل صلاحيات تطبيق معيّن.
السبب: التطبيق يملك صلاحية لا يحتاجها لوظيفته الأساسية.
البيانات المتأثرة: إعدادات الصلاحيات فقط.
الأثر المتوقع: تحسين الخصوصية، وقد تتوقف بعض الميزات التي تعتمد على هذه الصلاحية.
مستوى الخطورة: متوسط.
يتطلب التحقق: Face ID أو المصادقة الثنائية.
هل توافق؟ نعم / لا
```

## 10. سجل التدقيق

يجب أن يحتفظ النظام بسجل واضح لكل إجراء مهم.

### يجب أن يتضمن السجل

- وقت الإجراء.
- نوع الإجراء.
- سبب الإجراء.
- هل تمت الموافقة عليه أم رفضه.
- طريقة التحقق المستخدمة.
- النتيجة.

### يجب ألا يتضمن السجل

- كلمات المرور.
- رموز المصادقة.
- مفاتيح API.
- محتوى الرسائل الخاصة إلا إذا اختار المستخدم حفظها صراحة.

## 11. الوضع التلقائي

يمكن للنظام أن يحدد الوضع المناسب تلقائيًا بناءً على السياق، لكن التنفيذ يبقى حسب مستوى الخطورة.

### أوضاع مقترحة

- وضع الخصوصية العالية.
- وضع العمل.
- وضع السفر.
- وضع الاستخدام اليومي.
- وضع الطوارئ.
- وضع الفحص الأمني.

### قاعدة التنفيذ

النظام يمكنه تغيير التوصيات تلقائيًا، لكنه لا يغير إعدادات حساسة تلقائيًا دون موافقة.

## 12. متطلبات الأمان التقنية المستقبلية

عند بناء التطبيق فعليًا، يجب الالتزام بالآتي:

- استخدام تشفير للبيانات الحساسة محليًا.
- عدم تخزين أسرار في ملفات المشروع.
- استخدام مخزن أسرار آمن مثل Keychain أو Credential Manager عند توفره.
- طلب أقل صلاحيات ممكنة.
- فصل الأدوار بين القراءة والتنفيذ.
- إضافة اختبارات أمنية قبل أي إصدار.
- مراجعة التبعيات قبل تثبيتها.
- منع سكربتات التثبيت التلقائية غير الموثقة.

## 13. تعريف النجاح

يعتبر النظام ناجحًا عندما يحقق الآتي:

- يساعد المستخدم دون تعقيد.
- لا ينفذ إجراءات حساسة دون موافقة.
- يحمي الخصوصية بدل أن يجمع بيانات غير ضرورية.
- يوضح المخاطر بلغة مفهومة.
- يحترم سياسات التطبيقات والمنصات.
- يعطي المستخدم تحكمًا حقيقيًا وقابلًا للمراجعة.
